## Q.31 — Bedrock Custom Model Import
**問題:** 外部でファインチューニングされた Mistral-7B（SafeTensors 形式）を、SageMaker エンドポイント管理を避けて Bedrock のサーバーレス基盤で推論したい。Custom Model Import の正しいワークフローは？

- A. 重みを Bedrock の Models コンソールに直接アップロードし Model Customization で Anthropic 互換形式に変換
- **B. Mistral-7B の重みと設定ファイルを S3 にパッケージ化し、アーキテクチャが Bedrock でサポートされていることを確認し、Custom Model Import ジョブでサーバーレスのモデル ARN を作成する** ✅
- C. Bedrock Export 付き SageMaker Training Job で Model Registry から移行
- D. Bedrock Marketplace にサードパーティとして登録し VPC ピアリング

**正解: B** — Custom Model Import は自前の重み（Mistral, Llama など対応アーキテクチャ）を S3 から持ち込み、サーバーレスでホストする。S3 と Bedrock は同一リージョン、課金は Provisioned Throughput（Custom Model Units）。

## Q.32 — 主観的品質の評価（Human Evaluation）
**問題:** ファインチューニングした Claude 3 の 3 バージョンを比較し、最も「ブランド一貫性」「人間らしい」マーケティングコピーを出すものを判断したい。ROUGE/BLEU では捉えられない主観的ニュアンスを測る、最も統合された方法は？

- A. SageMaker＋カスタム React アプリ＋RDS で投票保存
- **B. Amazon Bedrock Model Evaluation の Human Evaluation ジョブで、Brand Tone / Engagement level 等のカスタム指標を使いマネージド UI で評価** ✅
- C. CloudWatch Evidently で本番 A/B テスト（CTR）
- D. Batch Inference ＋ LLM-as-a-judge（Claude 3 Opus）で採点

**正解: B** — 主観的・定性的品質はマネージドな Human Evaluation が最適。A は自作で非統合、C は定量指標、D は人間でなく LLM 採点。

## Q.34 — RLHF のマネージドワークフロー
**問題:** 静的データでは捉えにくい企業安全ガイドラインへのアライメントを RLHF で向上させたい。Bedrock の正しいマネージド RLHF 手順は？

- A. フルファインチューニング → ローカルで PPO → 再アップロード
- B. Model Customization で Continued Pre-training を選び Good/Bad 二値ラベルの報酬データを提供
- **C. ベースモデルを選び、Model Customization を RLHF 手法で構成し、プロンプトデータセットを提供、人間ワークフォースまたは事前学習済み報酬モデルで応答をランク付けしてポリシー最適化** ✅
- D. Bedrock Agent で Online Learning でリアルタイム重み更新（※存在しない）

**正解: C** — RLHF の本質は「プロンプト→応答生成→人間/報酬モデルでランク付け→強化学習でポリシー更新」。

## Q.35 — モデル蒸留（Distillation）
**問題:** Claude 3.5 Sonnet が高性能だが大量要約ワークロードでトークンコストが高い。Teacher-Student で Sonnet のロジックを模倣する小型高速モデルを作りたい。

- **A. SageMaker で Model Distillation を実行し、Sonnet の出力を正解ラベルとして Llama-3-8B など Student モデルをファインチューニング** ✅
- B. Weight Pruning で attention head の 50% を削除
- C. Student の Top-P / Top-K を上げて推論能力を模倣
- D. Bedrock コンソールで Model Quantization（FP16→INT4）

**正解: A** — Teacher-Student で小型モデルを作るのは Distillation。Pruning/Quantization は圧縮手法、Top-P/K は生成制御。

## Q.36 — 入出力のアーカイブ（Model Invocation Logging）
**問題:** 金融機関でプロンプトとモデル応答を 7 年間監査・アーカイブ。保存時暗号化、中央アカウント保存。最も運用負荷が少ない構成は？

- **A. Bedrock の Model Invocation Logging を有効化し、宛先をクロスアカウント S3 にし、S3 バケットポリシーと KMS キーで 7 年保持** ✅
- B. InvokeModel をラップする Lambda で DynamoDB に TTL 7 年で書き込み
- C. CloudTrail Data Events を CloudTrail Lake で保存
- D. EventBridge → Kinesis Data Firehose → 暗号化 S3

**正解: A** — Model Invocation Logging はプロンプト/応答の「中身」を記録するネイティブ機能。CloudTrail はメタデータのみ記録する点に注意。

## Q.37 — Query Expansion / Multi-Query
**問題:** RAG 法律アシスタントで、ベクトル検索が文字どおりすぎて関連文書を取りこぼす。Query Expansion（Multi-Query）の技術的実装は？

- A. k-NN の k を 5→100 に増やし reranker でフィルタ
- B. Hierarchical Chunking に変更
- **C. LLM で元クエリの 3〜5 バリエーションを生成し、各々でベクトル検索、得られたユニーク文書 ID を結合して生成に渡す** ✅
- D. OpenSearch Serverless で Lexical Overlap を有効化

**正解: C** — クエリを言い換えて複数検索→統合するのが Query Expansion。D は文字どおり一致を強め逆効果。

## Q.38 — Tensor Parallelism（巨大モデルの OOM）
**問題:** 70B Llama を p4d.24xlarge クラスターで学習中、バッチサイズ 1 でも OOM。ZeRO-3 は実装済み。追加すべき SMP 戦略は？

- **A. Tensor Parallelism を有効化し、単一ノード内の複数 GPU に個々のレイヤーと重みを分割** ✅
- B. Asynchronous SageMaker Training でメモリスパイク削減
- C. SageMaker Pipe Mode でデータローダーのメモリ削減
- D. SMDDP の Data Parallelism に切替えて GPU 総数を増やす

**正解: A** — バッチ 1 でも OOM ＝ モデル自体が GPU に乗らない → モデル並列（Tensor Parallelism）が必要。Data Parallel は各 GPU にフルモデル複製なので無効。

## Q.39 — Provisioned Throughput のライフサイクル
**問題:** Claude 3 Opus の PT を契約。コストが想定より高く、使っていないモデルバージョンに複数 Model Units が残存。PT のライフサイクル管理上の重要事実は？

- A. 48 時間リクエストがないと自動削除（※誤り）
- B. カスタムモデルの PT は先に S3 バケットを削除しないと削除不可（※誤り）
- **C. コミットメント（1か月/6か月）付き PT は期間終了まで削除できず、全期間分が請求される** ✅
- D. PT を Pause して課金停止可（※存在しない）

**正解: C** — コミット付き PT は途中解約しても残期間分の支払い義務あり。自動削除や一時停止はない。
